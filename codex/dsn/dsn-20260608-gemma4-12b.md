# Decision Support Note (DSN)

**ID:** DSN-20260608-01  
**Title:** Target Gemma 4 12B mxfp8 for unified any-to-any packs  
**Date:** 2026-06-08  
**Status:** Accepted  
**Evidence Status:** Quality signal on fault-code pack; broader validation still experimental
**Related Research Inbox Entry:** <none>

---

## Context
- The pack system is moving from text-only demos toward a generalist base with swappable LoRA "DLC" packs.
- The two exact MLX community candidates are `mlx-community/gemma-4-12B-bf16` and `mlx-community/gemma-4-12B-mxfp8`.
- Hugging Face metadata shows both exact candidates use `gemma4_unified`, task `any-to-any`, library `mlx`, base `google/gemma-4-12B`, and Apache-2.0 licensing. The mxfp8 repo is tagged 8-bit and has a much smaller local footprint.
- Because the project is macOS-centered, modality support should be explicit: `mlx-vlm` handles Gemma unified image/audio/video prompting, while `mlx-audio` handles dedicated TTS, STT, and STS workflows around the pack runtime.

## Options Considered
1. Use `mlx-community/gemma-4-12B-bf16` as the default
   - Pros: highest-fidelity weight path, useful for rank probing and reference comparisons.
   - Cons: larger local footprint and higher memory pressure for iterative pack work.
2. Use `mlx-community/gemma-4-12B-mxfp8` as the default
   - Pros: keeps unified any-to-any architecture, lowers local memory/disk pressure, better fit for on-demand pack swapping on Apple Silicon.
   - Cons: quantized rank probes may need fp16 fallback for pathological projections.
3. Switch to an instruction-tuned QAT variant now
   - Pros: likely better assistant behavior out of the box.
   - Cons: not one of the two base any-to-any repos under review; should be evaluated separately as a product-facing candidate.

## Decision
- Chosen option: Use `mlx-community/gemma-4-12B-mxfp8` as the default runtime base.
- Rationale: It preserves the unified any-to-any processor/model shape while making local iterative pack creation, loading, and routing more practical.
- Canonical decision record: `codex/decisions.md` ADR-0004.

## Consequences
- Immediate impacts: CLI examples default to Gemma 4 mxfp8, `packs` includes `mlx-vlm` and `mlx-audio`, loader auto-selection routes Gemma 4 unified bases through `mlx-vlm`, and LoRA manager discovers `language_model.model.layers`.
- Risks/unknowns: Real 12B pack training still needs hardware validation; instruction-tuned QAT variants may be better for the final assistant UX.
- Mitigations: Keep bf16 as a reference model, keep alpha/rank/size guardrails, test the manager against a synthetic Gemma unified shape, and evaluate QAT mxfp8 separately before changing the default.
- Validation gate: Promote broader quality claims only after reproducible held-out Gemma 4 bakeoffs show domain lift across more than the current fault-code pilot.

## Smoke Results
- `scripts/gemma4_smoke.py --metadata-only` confirmed mxfp8 and IT QAT mxfp8 are both about 11.77 GB; bf16 is about 22.31 GB and remains opt-in.
- `mlx-community/gemma-4-12B-mxfp8` loaded from local cache in 2.6-3.5 seconds, generated at about 30.6-31.0 tokens/sec for 8-token smoke prompts, and peaked at about 12.73 GB.
- The alpha-zero no-op pack probe passed on the actual Gemma mxfp8 model: output matched after export/reapply, and the exported pack was 3,585,584 bytes.
- Actual Gemma geometry includes 8 `use_k_eq_v` layers with no separate `v_proj` and 512-wide k slices; manager logic now skips missing per-block projections and uses block-local projection dimensions.
- Base mxfp8 output behaves like a base model rather than an assistant; IT QAT mxfp8 is the stronger product-facing candidate when prompts are rendered through the chat template.
- `mlx-community/gemma-4-12B-it-qat-mxfp8` with `--chat-template` loaded warm in about 2.8 seconds, peaked at about 12.75 GB, generated at about 25.9-50.0 tokens/sec on short prompts, and produced coherent assistant-style outputs.
- IndustryBench English pilot extraction produced 128 train and 32 held-out rows from `alibaba-multimodal-industrial-ai/IndustryBench` with clean Q/A training text and source metadata retained in JSON fields.
- A no-shrink heavy-profile 20-step IndustryBench pack on `mlx-community/gemma-4-12B-it-qat-mxfp8` exported and applied successfully: q/v rank 64, k rank 32, 91.75 MB. Held-out eval worsened PPL from 1125.20 to 1141.52 (+1.45%) and token accuracy from 0.28378 to 0.28342, while `max_logit_diff=3.0` confirmed the pack changed model behavior.
- `avneetsingla/industrial-fault-codes-sample` is a better first useful industrial training source than IndustryBench for the DLC-pack thesis: it has 3,000 English maintenance rows with explicit `brand`, `code`, `description`, and `solution` fields. License is `cc-by-nc-4.0`, so it is research/prototype data unless licensing is resolved.
- `scripts/fault_codes_extract.py` produced 2,400 train and 300 eval rows with prompt/answer/messages/text fields. The CLI now supports `--loss-mode answer` so the pack learns only assistant answer tokens while treating the fault-code prompt as context.
- The initial 10-step rank-16 answer-only fault-code smoke exported but did not change logits (`max_logit_diff=0.0`); direct tensor inspection showed adapter updates were mostly fp16 subnormal. LoRA train init was changed from `1/input_dim` to `1/sqrt(input_dim)` for the random `B` matrix while keeping zero `A`.
- With the fixed init, a 20-step rank-16 fault-code pack changed logits (`max_logit_diff=15.375`) and improved 64-row heldout answer-only PPL from 15.5308 to 15.4384 (-0.60%), with token accuracy unchanged.
- A larger fault-code sweep on the full 300-row heldout set shows this is a useful-pack path, not just mechanics:
  - Base: answer-only PPL 15.4316, token accuracy 0.6155.
  - Rank 16 / 100 steps: 27.10 MB, PPL 14.4647 (-6.27%), token accuracy 0.6197.
  - Rank 16 / 300 steps: 27.10 MB, PPL 11.6834 (-24.29%), token accuracy 0.6311.
  - Rank 32 / 100 steps: 54.16 MB, PPL 12.8972 (-16.42%), token accuracy 0.6259.
  - Rank 32 / 300 steps: 54.16 MB, PPL 8.5175 (-44.80%), token accuracy 0.6513.
- Generation checks on 8 heldout examples keep brand/code mention rates at 1.0/1.0. The rank-32 / 300-step pack improves solution keyword overlap from 0.2723 to 0.3619; rank-16 / 300 worsens it slightly to 0.2584 despite strong PPL gains.
- Best current useful-pack candidate: `fault-codes-gemma4-it-answer-r32-300`. Rank 16 / 300 is the best smaller pack by PPL-per-MB, but rank 32 / 300 is the first one that improves both answer-token eval and generated solution overlap.
